## 二、缺陷报告

### DEFECT-AUTH-001：居民端与医生端登录页面无角色校验

| 字段 | 内容 |
|---|---|
| 缺陷编号 | DEFECT-AUTH-001 |
| 缺陷标题 | 居民端与医生端登录页面无角色校验，居民账号可登录医生端、医生账号可登录患者端 |
| 缺陷所属模块 | 登录认证 |
| 适配角色 | 居民/医生 |
| 缺陷严重程度 | P1（严重） |
| 缺陷优先级 | 高 |
| 前置条件 | 后端服务已启动，数据库存在居民账号 zhangsan/123456 和医生账号 doctor/doctor |
| 复现步骤 | 1. 打开医生端页面 doctor.html；2. 将"账号"输入框内容改为 zhangsan，"密码"改为 123456；3. 点击"登 录"按钮；4. 观察页面跳转和数据显示 |
| 实际现象 | 登录成功，页面跳转至医生工作台，顶部显示"张医生"（硬编码），但咨询单列表加载的是全部数据，居民以医生身份操作系统 |
| 预期现象 | 前端在登录成功后应校验返回的 role 字段：若 role 为 patient 则禁止进入医生端并提示"该账号无医生权限"；若 role 为 doctor 则禁止进入患者端 |
| 关联用例编号 | ROLE-Isolation-001, ROLE-Isolation-002 |
| 缺陷状态 | 新建 |

### DEFECT-CHECK-002：AI体质辨识结果与舌象图片内容无关

| 字段 | 内容 |
|---|---|
| 缺陷编号 | DEFECT-CHECK-002 |
| 缺陷标题 | AI体质辨识结果仅由年龄和性别取模决定，与上传的舌象图片内容无关 |
| 缺陷所属模块 | 舌诊辨识（AI算法） |
| 适配角色 | 居民 |
| 缺陷严重程度 | P1（严重） |
| 缺陷优先级 | 高 |
| 前置条件 | 居民已登录且档案中 age=45, gender=male |
| 复现步骤 | 1. 点击"舌诊辨识"Tab；2. 上传一张明显偏红色的舌象正面照；3. 点击"开始 AI 体质辨识"按钮；4. 记录辨识结果；5. 重新上传一张明显偏白色的舌象正面照；6. 再次点击"开始 AI 体质辨识"按钮；7. 对比两次结果 |
| 实际现象 | 两次辨识结果完全相同（均为"痰湿质"，confidence=85.0），因为 `mockAiAnalysis` 函数中 `const idx = (age + (gender === 'male' ? 0 : 3)) % constitutions.length`，结果仅由 age=45 和 gender=male 决定，与图片内容无关 |
| 预期现象 | 不同舌象图片应产生不同的辨识结果，AI 分析应基于图像特征而非仅依赖用户档案的年龄和性别字段 |
| 关联用例编号 | CHECK-User-001 |
| 缺陷状态 | 新建 |

### DEFECT-SEC-005：登录接口使用明文密码存储与比对

| 字段 | 内容 |
|---|---|
| 缺陷编号 | DEFECT-SEC-005 |
| 缺陷标题 | 登录接口使用明文密码比对，password_hash 字段实际存储明文而非哈希值 |
| 缺陷所属模块 | 登录认证（安全） |
| 适配角色 | 居民/医生 |
| 缺陷严重程度 | P1（严重） |
| 缺陷优先级 | 高 |
| 前置条件 | 数据库已初始化，种子数据已导入 |
| 复现步骤 | 1. 查看数据库 user_accounts 表，确认 password_hash 字段值为"123456"（明文）；2. 查看 server.js 中登录 SQL：`WHERE username = ? AND password_hash = ?`，直接将用户输入的明文密码与数据库明文比对；3. 使用 SQL 注入测试：在密码框输入 `' OR '1'='1`，观察是否绕过验证 |
| 实际现象 | password_hash 字段存储的是明文密码（如"123456"、"doctor"），而非 SHA256/Bcrypt 等哈希值；登录接口直接明文比对，存在数据泄露风险；SQL 注入虽被参数化查询防御，但明文存储仍违反安全规范 |
| 预期现象 | 数据库应存储密码的哈希值（如 Bcrypt hash），登录时将用户输入密码哈希后与数据库比对；字段名 password_hash 应名副其实 |
| 关联用例编号 | LOGIN-User-001, LOGIN-Doctor-001 |
| 缺陷状态 | 新建 |
